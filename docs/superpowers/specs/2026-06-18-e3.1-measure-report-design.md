# E3.1 — Emit FHIR MeasureReport — Design

Date: 2026-06-18
Issue: #89 (sub-issue of epic E3 / #73 — eCQM artifact completeness)
Status: Approved (brainstorm) — pending spec review
Branch: TBD (`feat/issue-89-measure-report`)

## 1) Goal

Emit FHIR R4 `MeasureReport` resources (individual + summary) from a **completed run**, so a
run's compliance results are expressible as the standard eCQM reporting artifact. Population counts
reconcile exactly with the persisted `outcomes`, and the resources are structurally conformant to
the FHIR R4 `MeasureReport` StructureDefinition.

This is the first E3 deliverable: today we export the *measure definition* (Measure/Library/ValueSet
MAT XML via `src/fhir/mat-export.ts`) but **no MeasureReport** (the *result-of-evaluation* artifact).

## 2) Current state (what we reuse, not rebuild)

- **Outcomes are persisted per run** (`outcomes`: `employee_id`, `measure_version_id`,
  `evaluation_period`, `status` ∈ {COMPLIANT, DUE_SOON, OVERDUE, MISSING_DATA, EXCLUDED},
  `evidence_json`). The report is built **from these rows** — no re-evaluation, no engine change.
- **Run + measure metadata:** `runs` carries `measurement_period_start/end` and `scope_id` (the
  measure-version UUID for MEASURE runs); `measure_versions` → `measures` gives name/version/policy_ref.
- **`src/fhir/mat-export.ts`** is the sibling pattern to follow (a pure FHIR-builder module + a route),
  and establishes the **measure canonical-URL convention** the MeasureReport's `measure` field reuses
  for consistency.
- **Deps:** `cql-execution`/`cql-exec-fhir` only — **no `fqm-execution`**. We construct the
  MeasureReport JSON directly (no new dependency).

## 3) Scope

In scope:
- A pure builder module `src/fhir/measure-report.ts` producing FHIR R4 `MeasureReport` JSON.
- Summary report (aggregate population counts for a run's measure) + individual reports (one per
  subject) + a Bundle assembler.
- A run-scoped read endpoint serving them.
- JVM-free structural conformance validation + reconciliation tests.

Out of scope (YAGNI / separate issues):
- Real value-set expansion (#90), QRDA III (#91), QI-Core profiles (#92).
- Persisting MeasureReports (generation-on-read only).
- `evaluatedResource`/define-level evidence inside individual reports (that data already lives in
  `outcomes.evidence_json`; linking it is a later enhancement).
- Non-proportion measure scoring models (CMS122 is value-driven; for E3.1 every runnable measure is
  reported with the proportion model below — see Risks).

## 4) Design

### 4.1 Population mapping (the reconciliation contract)

Proportion model. For a run's outcomes (counts per bucket):
- **initial-population (IPP)** = total evaluated (all outcomes for the run).
- **denominator-exclusion (DENEX)** = `EXCLUDED` count.
- **denominator (DENOM)** = IPP − DENEX.
- **numerator (NUMER)** = `COMPLIANT` count.
- **measureScore** (summary only) = NUMER / DENOM (Quantity, `value` in [0,1]; DENOM=0 → score omitted).
- `DUE_SOON`, `OVERDUE`, `MISSING_DATA` are denominator gaps (in DENOM, not NUMER).

This matches the existing `/runs` pass-rate exactly, so the report reconciles 1:1 with `outcomes`.

### 4.2 Module `src/fhir/measure-report.ts` (pure builders)

- `buildSummaryMeasureReport(run, measure, outcomes): MeasureReport`
  - `resourceType: "MeasureReport"`, `status: "complete"`, `type: "summary"`,
    `measure: <canonical>` (MAT-export convention), `period: { start, end }` (run measurement period),
    `improvementNotation: { coding: [{ system: ".../measure-improvement-notation", code: "increase" }] }`,
    one `group` with `population[]` (each `{ code: { coding:[{ system: "http://terminology.hl7.org/CodeSystem/measure-population", code: <IPP|DENOM|DENEX|NUMER> }]}, count }`) and `measureScore: { value }`.
- `buildIndividualMeasureReport(outcome, run, measure): MeasureReport`
  - same header but `type: "individual"`, `subject: { reference: "Patient/<employee fhir id or external id>" }`,
    `group.population[]` counts are 0/1 reflecting that subject's membership
    (COMPLIANT → IPP=1,DENOM=1,NUMER=1,DENEX=0; EXCLUDED → IPP=1,DENEX=1,DENOM=0,NUMER=0;
    DUE_SOON/OVERDUE/MISSING_DATA → IPP=1,DENOM=1,NUMER=0,DENEX=0). No `measureScore` on individual.
- `buildMeasureReportBundle(run, measure, outcomes): Bundle`
  - `type: "collection"`, entries = summary first, then one individual per outcome.

Builders are **pure** (data in → JSON out); counts derive from the same `outcomes` array, so
reconciliation holds by construction.

### 4.3 Endpoint

`GET /api/runs/{runId}/measure-report?type=summary|individual|bundle` (default `summary`):
- Resolves the run, its outcomes, and the measure version (404 if the run doesn't exist; 409/422 if
  the run isn't a single-measure run — see Risks); role-gated like other run reads.
- `summary` → the summary `MeasureReport`; `individual` or `bundle` → the collection Bundle.
- Content-Type `application/fhir+json`.

### 4.4 Validation (JVM-free, structural)

Hand-rolled conformance assertions (no new dep, no JVM):
- Required elements present: `resourceType`, `status` ("complete"), `type` (∈ R4 valueset), `measure`,
  `period.start`/`period.end`; `subject.reference` present on individual, absent on summary.
- `group.population[].code` codings use the exact `measure-population` system + valid codes; `count` is
  a non-negative integer.
- Summary `measureScore.value` ∈ [0,1] and equals NUMER/DENOM; internal consistency: IPP = DENOM + DENEX,
  NUMER ≤ DENOM.

## 5) Testing

`src/fhir/measure-report.test.ts`:
1. **Reconciliation (builders):** given a synthetic run + a known bucket distribution (e.g. 78/8/9/2/3
   for COMPLIANT/DUE_SOON/OVERDUE/MISSING_DATA/EXCLUDED), assert IPP=100, DENEX=3, DENOM=97, NUMER=78,
   measureScore≈0.804, and that summing individual reports' population memberships equals the summary counts.
2. **Structural conformance:** assert §4.4 invariants for both summary + individual reports.
3. **Edge cases:** all-EXCLUDED run (DENOM=0 → measureScore omitted, no divide-by-zero); single-subject run.

`src/routes/runs.test.ts` (or a sibling): a route test that runs/loads a seeded or synthetic run, calls
`GET /api/runs/{id}/measure-report?type=summary`, and asserts the served counts equal
`GET /api/runs/{id}/outcomes` aggregated by bucket (the live reconciliation guarantee).

## 6) Acceptance criteria (maps to #89)

- Individual + summary `MeasureReport` resources generated from a completed run.
- Population counts reconcile with persisted `outcomes` (verified by test #1 + the route test).
- Resources are structurally conformant to FHIR R4 `MeasureReport` (verified by test #2; JVM-free).
- `pnpm typecheck` + `pnpm test` green; no new dependency; engine untouched.

## 7) Risks / open points

- **Non-proportion measures (CMS122).** CMS122 is value-driven (HbA1c >9% = poor control), not a
  classic proportion measure, yet its outcomes still bucket into the 5 statuses. For E3.1 it is reported
  with the same proportion model (COMPLIANT=NUMER etc.), which is a faithful *operational* rollup even if
  not the canonical CMS122 numerator semantics. Documented as a known simplification; true per-measure
  scoring (`scoring`/`measure-observation`) is future work. **Decision: proportion model for all, noted.**
- **Subject reference.** Individuals reference `Patient/<id>`. Employees carry `fhir_patient_id` (nullable)
  + `external_id`; the builder uses `fhir_patient_id` when present, else `external_id` (stable, synthetic).
- **`measure` canonical.** Reuse `mat-export.ts`'s measure-URL convention so the MeasureReport's `measure`
  matches the exported Measure resource; confirm the exact string during implementation.
- **Run scope.** Only single-measure runs (MEASURE/CASE/EMPLOYEE/SITE for one measure) produce a coherent
  single-measure report. For ALL_PROGRAMS runs (multiple measures), the endpoint returns 422 with a clear
  message (or, future, one report group per measure). **Decision for E3.1: single-measure runs only; 422 otherwise.**
