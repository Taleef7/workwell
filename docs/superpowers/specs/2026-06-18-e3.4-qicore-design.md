# E3.4 — QI-Core Profile Alignment (structural) — Design

Date: 2026-06-18
Issue: #92 (sub-issue of epic E3 / #73 — eCQM artifact completeness; **stretch**)
Status: Design (autonomous — best-judgment, no clarifying-question gate per maintainer instruction)
Branch: `feat/issue-92-qicore-profiles`

## 1) Goal

Align the synthetic FHIR R4 bundles to **QI-Core** so the evaluated resources declare QI-Core profile
conformance (`meta.profile`) and carry the structurally-required elements — **structural alignment,
JVM-free, no validator dependency** (consistent with ADR-009), not official IG/validator validation.

## 2) Approach decision (recorded)

#92's acceptance says "validate against the relevant QI-Core profiles." Real validation needs the HL7
FHIR validator + the QI-Core IG (a JVM tool) or a new JS FHIR-validator dependency — both excluded by
the JVM-free / no-new-dep stack (ADR-008/009). So this delivers **structural QI-Core alignment**: stamp
`meta.profile` with the QI-Core canonicals and assert the QI-Core-required elements structurally. This is
the same honest-stub posture as the FHIR MeasureReport (E3.1) and QRDA III (E3.3); the conformance matrix
states the level. Full IG validation remains future work (would need an approved validator dep).

## 3) Scope

In scope:
- `backend-ts/src/engine/synthetic/fhir-bundle-builder.ts` — add `meta: { profile: [<qicore-url>] }` to
  every emitted resource (Patient, enrollment/waiver Condition, Observation, Immunization, Procedure) via
  a `QICORE_PROFILES` map keyed by `resourceType`.
- A test asserting each resource in a synthetic bundle carries the expected QI-Core profile + its
  QI-Core-required structural elements.
- `docs/STANDARDS_CONFORMANCE.md` — a QI-Core row.
- Docs (MEASURES / ARCHITECTURE / JOURNAL).

Out of scope (recorded):
- Official QI-Core IG / FHIR-validator validation (needs a JVM or a new dep — separate approved decision).
- Profile *slug* version-exactness — the well-known QI-Core canonicals are used; representative for a
  structural stub (documented).
- Any change to the CQL, the ELM, the engine, or the stores. No schema change.

## 4) Design

- `QICORE_PROFILES: Record<string, string>` (base `http://hl7.org/fhir/us/qicore/StructureDefinition/`):
  `Patient → qicore-patient`, `Condition → qicore-condition`, `Procedure → qicore-procedure`,
  `Immunization → qicore-immunization`, `Observation → qicore-observation-clinical-result`.
- Stamp `meta: { profile: [QICORE_PROFILES[resourceType]] }` on each resource the builder emits (the
  `condition()` helper + the Patient + the three event resources).
- **Evaluation is unaffected:** `meta.profile` is metadata; the CQL retrieves match on code/system, which
  are unchanged. So the 10-measure golden-parity outcomes are byte-identical (the existing engine contract
  test is the guard).

## 5) Testing

`backend-ts/src/engine/synthetic/fhir-bundle-builder.test.ts` (extend — it exists):
- For a synthetic bundle (enrolled, with an event), assert every entry's `resource.meta.profile[0]` equals
  the expected QI-Core canonical for its `resourceType`.
- Assert the QI-Core-required structural elements are present per resource: Patient (`id`, `name`),
  Condition (`clinicalStatus`, `verificationStatus`, `code`, `subject`), Observation (`status`, `code`,
  `subject`), Procedure (`status`, `code`, `subject`), Immunization (`status`, `vaccineCode`, `patient`).
- The existing bundle-builder assertions + the full suite (esp. `cql-execution-engine.test.ts` 10-measure
  parity) stay green — proving `meta.profile` didn't change evaluation.

## 6) Acceptance (maps to #92, structural interpretation)

- Synthetic bundles declare QI-Core profile conformance (`meta.profile`) + carry the required elements,
  asserted structurally. ✓
- Conformance documented in `docs/STANDARDS_CONFORMANCE.md` (QI-Core row, level = structural alignment).
- `pnpm typecheck` + `pnpm test` green; golden parity unchanged; no new dep; no schema change.

## 7) Risks / open points

- **Structural, not IG-validated** (recorded; ADR-009 posture). Real QI-Core validation needs a validator
  dep/JVM = a separate approved decision. The matrix states the level honestly.
- **Profile slug exactness** — the well-known QI-Core canonicals are representative; exact version/slug
  alignment (e.g. QI-Core 4.1.1 vs 6.x condition split) is deferred. Documented.
- **Blast radius** — the synthetic bundle builder underpins all 10 measures + their golden tests; adding
  `meta.profile` is additive metadata (no code/system change), so parity holds — guarded by the existing
  10-measure contract test, which must stay green.
