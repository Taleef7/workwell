# E3.2 — Real Value-Set Expansion — Design

Date: 2026-06-18
Issue: #90 (sub-issue of epic E3 / #73 — eCQM artifact completeness)
Status: Approved (brainstorm) — pending spec review
Branch: `feat/issue-90-valueset-expansion`

## 1) Goal

Replace the inline-code matching workaround (documented in `docs/MEASURES.md` Implementation Notes)
with **real value-set expansion**: ≥1 value set resolves from a real expansion source and *drives
evaluation* through a populated CQL `CodeService` — while the existing inline path keeps working,
config-selectable, so the demo is unaffected. The expansion source is **local** (store-backed)
behind a pluggable `ValueSetResolver` port, so a live VSAC adapter is a future drop-in.

## 2) Current state

- **The engine does no value-set expansion today.** `cql-execution-engine.ts` constructs
  `new cql.CodeService({})` — an *empty* CodeService — so CQL value-set retrieves would match nothing.
- **Measures inline-match instead.** `measures/audiogram.cql` *declares*
  `valueset "Audiogram Procedures": 'urn:workwell:vs:audiogram-procedures'` but never uses a
  value-set retrieve; it inline-matches:
  `[Procedure] P where exists(P.code.coding C where C.system = 'urn:workwell:vs:audiogram-procedures' and C.code = 'audiogram-procedure')`.
- **Synthetic data shape:** `SyntheticFhirBundleBuilder` stamps the qualifying resource with
  `coding = { system: <the value-set OID>, code: <binding.event.code> }` (for audiogram:
  `system: 'urn:workwell:vs:audiogram-procedures', code: 'audiogram-procedure'`).
- **Storage already exists:** `ValueSetStore` returns `ValueSetRecord.codes: CodeEntry[]`
  (`{ code, display, system }`), seeded `resolution_status=RESOLVED`. The `value_sets` table already
  has `codes_json`/`expansion_hash`/`resolution_status` — **no schema change needed** for expansion.

## 3) Scope

In scope (the approved "Both"):
- A `ValueSetResolver` port + a local **store-backed** adapter + a `buildCodeService(...)` that returns a
  *populated* `cql.CodeService`.
- An engine **config seam** selecting **inline mode** (empty CodeService, today's behavior — default)
  vs **expansion mode** (populated CodeService).
- **Convert one live measure (audiogram):** a value-set-retrieve CQL variant compiled to a second ELM,
  used in expansion mode; the inline ELM stays for inline mode.
- A focused **value-set-retrieve fixture** proving expansion drives a retrieve by membership.
- Seed the `urn:workwell:vs:audiogram-procedures` value set with the member code the synthetic data uses.

Out of scope (YAGNI / separate work):
- A live VSAC HTTP resolver (the port makes it a future drop-in; needs a UMLS key + network).
- Converting the other 9 measures (audiogram is the proof; the rest stay inline).
- Replacing the synthetic `system = VS-OID` convention with real CPT/LOINC value sets (would touch the
  bundle builder + every measure; a later refinement).
- Any DDL / schema migration (stop-and-ask; not needed here).

## 4) Design

### 4.1 `ValueSetResolver` port + local adapter — `src/engine/cql/value-set-resolver.ts`
- `interface ValueSetResolver { expand(valueSetUrl: string): Promise<CqlCode[]> }` where a `CqlCode` is
  `{ code: string; system: string }` (display optional).
- `StoreValueSetResolver implements ValueSetResolver` — reads `ValueSetStore` (`getById` keyed by the
  value set's OID/URL, or a `getByUrl` lookup) and maps `CodeEntry[] → CqlCode[]`. Returns `[]` for an
  unknown/unresolved value set (the retrieve then matches nothing — explicit, never guesses).
- `buildCodeService(resolver, valueSetUrls): Promise<cql.CodeService>` — resolves each referenced value
  set and constructs a `cql.CodeService` populated with `{ <valueSetOid>: [{ code, system }, …] }`
  (exact `cql-execution` CodeService shape confirmed at plan time). The referenced value-set URLs come
  from the measure's declared valuesets (registry/binding metadata).

### 4.2 Engine config seam — `cql-execution-engine.ts`
- A mode flag, `WORKWELL_VALUESET_EXPANSION` (default `false` = inline), surfaced through the engine
  (constructor option, env-read at the worker boundary like other config). 
- **inline mode:** unchanged — `new cql.CodeService({})` + the inline ELM. Default; the live demo and all
  10 measures are byte-for-byte unaffected.
- **expansion mode:** `buildCodeService(resolver, …)` populates the CodeService; the converted measure
  uses its value-set ELM.

### 4.3 Convert audiogram (dual-mode, parity-preserving)
- New `measures/audiogram_vs.cql` (or a variant): identical logic but the procedure retrieve becomes
  `[Procedure: "Audiogram Procedures"]` (a true value-set retrieve), keeping every define + outcome
  mapping identical. Compiled to its own ELM via the existing `compile-measures` path.
- `measure-registry.ts` maps `audiogram` → inline ELM (inline mode) | value-set ELM (expansion mode).
- The other measures map to a single (inline) ELM in both modes — a populated CodeService doesn't change
  inline-matching measures (they don't use value-set retrieves), so they stay green in expansion mode too.

### 4.4 Value-set seeding (data, not schema)
- Ensure `urn:workwell:vs:audiogram-procedures` is seeded (via the existing `seedValueSet` path) with a
  member `{ system: 'urn:workwell:vs:audiogram-procedures', code: 'audiogram-procedure' }` — matching what
  the bundle builder stamps — so the expansion-mode retrieve matches the synthetic Procedure and parity holds.

## 5) Testing

- **Resolver unit** (`value-set-resolver.test.ts`): a `StoreValueSetResolver` over a seeded store expands
  the audiogram VS to the expected `CqlCode[]`; unknown URL → `[]`; `buildCodeService` produces a CodeService
  whose entries match.
- **Value-set-retrieve fixture:** a minimal CQL library with a `[Procedure: "X"]` retrieve, compiled to ELM,
  evaluated with (a) an empty CodeService → no match, (b) the populated CodeService → match. Proves expansion
  *drives* the retrieve (not inline literals).
- **Cross-mode golden parity (audiogram):** for all 4 `spike/synthetic/audiogram/*` scenarios, assert the
  outcome in **inline mode == expansion mode** (and == the existing golden). This is the safety net that the
  conversion is behavior-preserving.
- **Full suite stays green** in the default (inline) configuration — the existing 459 tests are untouched.

## 6) Acceptance criteria (maps to #90)

- ≥1 value set (`Audiogram Procedures`) resolves from a real (store-backed) expansion source and **drives
  evaluation** (the audiogram value-set ELM's retrieve matches via the populated CodeService).
- The synthetic/inline path **keeps working alongside, config-selectable** (`WORKWELL_VALUESET_EXPANSION`
  default off → today's behavior; on → expansion path), proven by cross-mode golden parity.
- `pnpm typecheck` + `pnpm test` green; no new dependency; no schema change; the `ValueSetResolver` port
  leaves a clean seam for a future VSAC adapter.

## 7) Risks / open points

- **Synthetic codes use `system = the value-set OID`.** The expansion is "real" (member codes drive the
  retrieve via the CodeService) but the code *system* is the WorkWell VS OID, not a real terminology
  (CPT/LOINC). This is parity-preserving and sufficient for the proof; migrating the synthetic data +
  value sets to real terminologies is deferred (would touch the bundle builder + all measures). **Recorded.**
- **`cql-execution` CodeService shape** (how it keys value-set OID → codes, and how the ELM retrieve's
  `valueset` localId binds to it) will be confirmed against the library at plan time; the fixture test
  de-risks it before the audiogram conversion.
- **Dual ELM for audiogram** adds one compiled artifact + a registry branch; the cross-mode parity test
  guards against drift between the two.
- **Config default is off** (inline) so the deployed demo behavior is unchanged; expansion mode is opt-in
  until the other measures are converted in later work.
- **No DDL.** If implementation reveals a needed schema change, STOP and ask (owner-managed schema).
