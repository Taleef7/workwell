# E3.3 — QRDA III Stub + Standards-Conformance Matrix — Design

Date: 2026-06-18
Issue: #91 (sub-issue of epic E3 / #73 — eCQM artifact completeness)
Status: Design (autonomous — best-judgment, no clarifying-question gate per maintainer instruction)
Branch: `feat/issue-91-qrda3`

## 1) Goal

Emit a **QRDA Category III** (aggregate quality report) **stub** — a well-formed CDA XML document for a
completed single-measure run's aggregate results — and commit a **standards-conformance matrix** under
`docs/` mapping every artifact WorkWell emits (CQL / ELM / ValueSet / FHIR MeasureReport / MAT export /
QRDA III) against the eCQM standard, with an honest conformance level for each.

## 2) Current state (what we reuse)

- **Aggregate counts already exist:** `src/fhir/measure-report.ts` → `countPopulations(outcomes)` returns
  `{ ipp, denom, denex, numer }` (the proportion model from E3.1). QRDA III reports the *same* aggregate
  counts + measure score, in CDA form — so we **reuse `countPopulations`** (DRY; same reconciliation).
- **Hand-built-XML precedent:** `src/fhir/mat-export.ts` builds FHIR R4 XML by construction (a small
  `Xml` emitter, balanced by construction). QRDA III is also XML (HL7 CDA R2); same approach, no new dep.
- **Run-scoped read endpoints** are established (`/api/runs/:id/measure-report`): resolve run + outcomes,
  derive the single measure, serve. QRDA III mirrors this.
- No schema change. No new dependency.

## 3) Scope

In scope:
- `src/fhir/qrda3-export.ts` — `buildQrda3Document(run, measureId, outcomes): string` returning a
  well-formed QRDA III CDA XML document (header + a Measure Section + a Measure Reference and Results
  organizer with the aggregate population `observation`/`aggregateCount`s + the reporting period).
- A run-scoped endpoint `GET /api/runs/{runId}/qrda?format=xml` (`application/xml`), single-measure
  (422 multi-measure, 404 unknown run) — mirroring the measure-report branch.
- `docs/STANDARDS_CONFORMANCE.md` — the conformance matrix.

Out of scope (recorded):
- Full QRDA III IG / Schematron conformance validation (this is a **stub** — structurally representative,
  well-formed, not IG-validated; no XML-validator dependency, consistent with the JVM-free/no-new-dep stack).
- Multi-measure aggregate QRDA III (one measure section for the run's measure; multi-measure is future).
- QRDA I (patient-level). QI-Core profiles (#92).

## 4) Design

### 4.1 `src/fhir/qrda3-export.ts`
- A small balanced-by-construction XML emitter (local; or factor the `Xml` helper out of `mat-export.ts`
  into a shared `src/fhir/xml.ts` if clean — decide at implementation, default: local to avoid churn).
- `buildQrda3Document(run, measureId, outcomes)`:
  - **Header:** `<?xml?>`, `ClinicalDocument` with `typeId` (root `2.16.840.1.113883.1.3`, ext
    `POCD_HD000040`), the **QRDA Category III** `templateId` (`2.16.840.1.113883.10.20.27.1.1` +
    the QDM/CMS companion `…27.1.2`), a document `id`, `code` LOINC `55184-6` ("QRDA Calculated Summary
    Report"), `title`, `effectiveTime` (now), and the reporting period from the run's
    `measurementPeriodStart/End`.
  - **Body:** `component/structuredBody` → a **Measure Section** (`templateId 2.16.840.1.113883.10.20.27.2.1`,
    section `code` LOINC `55186-1`) → a **Measure Reference and Results** organizer
    (`templateId 2.16.840.1.113883.10.20.27.3.1`) referencing `urn:workwell:measure:<measureId>` and
    carrying population components: an `observation` per population (IPP/DENOM/DENEX/NUMER) with the
    HQMF population-type code and an `<value xsi:type="INT" value="<count>"/>` (`aggregateCount`), plus a
    performance-rate value (`numer/denom`).
  - Counts come from `countPopulations(outcomes)`.
- Template/OID strings are the well-known QRDA III IG identifiers; the document is a **stub** — the
  conformance matrix states the level (well-formed + structurally representative, not IG-validated).

### 4.2 Endpoint (`src/routes/runs.ts`)
`GET /api/runs/{runId}/qrda?format=xml`: resolve run (404 if null), list outcomes, derive distinct
`measureId` (422 if ≠ 1 — reuse the measure-report guard), build the document, return
`new Response(xml, { headers: { "content-type": "application/xml" } })`. `format` other than `xml`/absent
→ 400 (only XML is defined for QRDA). Mirror the existing measure-report branch placement (before the
`:id` detail branch).

### 4.3 `docs/STANDARDS_CONFORMANCE.md`
A table: **Artifact | Standard | What WorkWell emits | Conformance level | Notes**, with rows for:
- CQL (authored measures) — HL7 CQL 1.x — *authored, compiles*.
- ELM (compiled) — HL7 ELM — *build-time compiled, JVM-free; runtime-executed*.
- ValueSet expansion — FHIR ValueSet / VSAC — *store-backed expansion via `ValueSetResolver` (E3.2); VSAC-ready, synthetic codes today*.
- FHIR MeasureReport — FHIR R4 — *summary + individual; structurally conformant; counts reconcile (E3.1)*.
- MAT export — HQMF-adjacent (Measure/Library/ValueSet FHIR bundle) — *MAT-compatible export*.
- QRDA III — HL7 QRDA Category III — **stub** — *well-formed, structurally representative; not IG/Schematron-validated*.

## 6) Acceptance (maps to #91)

- A QRDA III aggregate export stub produces a **well-formed document** (verified structurally; balanced
  by construction). ✓
- A **standards-conformance matrix** is committed under `docs/STANDARDS_CONFORMANCE.md`. ✓
- `pnpm typecheck` + `pnpm test` green; no new dependency; no schema change.

## 7) Risks / open points

- **Stub, not IG-validated.** The document is well-formed and structurally representative of QRDA III but
  is **not** validated against the QRDA III Implementation Guide / Schematron (no XML-validator dep; JVM-free).
  The conformance matrix states this plainly. Full IG validation is future work.
- **TemplateId/OID accuracy** — the well-known QRDA III IG OIDs are used; exact versions/extensions will be
  set from the current IG at implementation. Since the deliverable is a *stub*, structural representativeness
  (not version-exact IG conformance) is the bar.
- **Well-formedness is asserted structurally** (substring/shape + balanced-by-construction emitter), not via
  an XML parser (no parser dependency). Acceptable for a stub; a parser-backed assertion is a future option.
- **Single-measure only** (one measure section); multi-measure aggregate QRDA III deferred.
