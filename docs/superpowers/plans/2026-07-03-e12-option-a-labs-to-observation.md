# Plan: Option A — re-point the 4 lab/vital measures from `[Procedure]` to `[Observation]`

**Status:** Planned / **recommend DEFER** (see §6). Working file, not yet executed.
**Context:** E12 PR-2b (merged, PR #234) ships a WebChart→FHIR adapter whose normalizer *synthesizes a
`Procedure`* from a real WebChart lab `Observation` because four measures — `diabetes_hba1c`,
`cholesterol_ldl`, `hypertension`, `obesity_bmi` — retrieve `[Procedure]` in their CQL even though real
WebChart records these labs/vitals as `Observation`s. "Option A" is the standards-correct end state:
re-point those four measures to retrieve `[Observation]`, retiring the synthesis workaround.

## 1. Goal

Real WebChart lab/vital `Observation`s evaluate these four measures **directly** (no synthesized
`Procedure`), matching real-world FHIR resource typing. Functionally the outcomes are unchanged — the
measures stay recency-based (days since last event); only the resource type + date field change
(`[Procedure].performed` → `[Observation].effective`). `cms122` is already `[Observation]`-based and is
the reference pattern — but it is **value-based** (reads `.value`, no recency window, no DUE_SOON), so its
config/builder path is NOT directly reusable for these recency measures.

## 2. Honest scope / risk (why this is bigger than it looks)

This is **not** a simple type flip. It touches measure semantics, the CQL codegen, ELM compilation, and
16 checked-in golden fixtures — ~31 files. Two hard couplings make it risky:

- **`exam-config.ts` misroute (critical).** `deriveExamConfig` branches on `event.type === "observation"`
  and applies **value-based** logic (sets `observationValue`, `daysSinceLastExam: 30`). Flipping the YAML
  `type` to `observation` would route these four **recency** measures into the value path → the builder
  emits a `valueQuantity`, and their COMPLIANT/DUE_SOON/OVERDUE **recency windows stop being exercised
  entirely**. Must add a way to distinguish "observation-recency" from "observation-value" (cms122).
- **CQL codegen hard-throws.** `cholesterol_ldl` + `hypertension` are rule-driven (`rule: windowed-recency`
  in their YAML) → generated by `scripts/gen-cql.mjs`. `generate-cql.ts` `windowedRecency()` **throws**
  `windowed-recency (E11.1) requires event.type=procedure` and hardcodes `[Procedure] P … .performed`. So
  Option A additionally requires generalizing the codegen to emit an `[Observation] … .effective` variant.
  (`diabetes_hba1c` / `obesity_bmi` are hand-written CQL only — simpler.)

Changing the canonical compliance semantics of **4 live measures** + regenerating ELM + golden data is
the kind of change where a mistake silently alters outcomes. It is reversible (revert the PR), but the
verification surface is large.

## 3. The design decision to make first

How to distinguish **observation-recency** (these 4) from **observation-value** (cms122) so `exam-config`
and the codegen route correctly. Options:
- **(a) Presence of a value rule:** treat `event.type === "observation"` + no value threshold ⇒ recency;
  with a value threshold ⇒ value. Gate `exam-config.ts:57` on that.
- **(b) An explicit binding flag:** add `event.recency: true` (or a `valueBased` flag) to the YAML/bindings
  and branch on it in both `exam-config.ts` and the builder. More explicit, additive to the binding type.

**Recommendation: (b)** — an explicit flag is clearer and less magical than inferring from value-rule
presence, and it makes the codegen guard easy to relax precisely.

## 4. Execution sequence (once the §3 decision is made)

1. **Design** the recency-vs-value discriminator (§3b: add the binding flag; extend `MeasureBinding`/YAML).
2. **CQL (hand-written):** `measures/{diabetes_hba1c,obesity_bmi}.cql` — `[Procedure]`→`[Observation]`,
   `performed`→`effective` (both the `sort by` and the trailing `.performed`).
3. **Codegen:** generalize `src/engine/cql/codegen/generate-cql.ts` `windowedRecency()` to emit an
   observation retrieve/`effective` when the binding is observation-recency; relax the `=procedure` guard.
   Regenerate `measures/generated/{cholesterol_ldl,hypertension}.cql` via `node --import tsx scripts/gen-cql.mjs`.
4. **YAML:** `measures/{diabetes_hba1c,cholesterol_ldl,hypertension,obesity_bmi}.yaml` — `type: observation`
   (+ the recency flag from step 1).
5. **Bindings:** `node scripts/gen-measure-bindings.mjs` → regenerates `measure-bindings.ts`.
6. **ELM:** `node scripts/compile-measures.mjs` → regenerates the 4 `*.elm.json` + `cql-resources.json`.
7. **Builder:** `src/engine/synthetic/fhir-bundle-builder.ts` — add an observation-recency emit branch in
   the recency block (Observation, `status:"final"`, `effectiveDateTime`, **no** `valueQuantity`).
8. **exam-config:** `src/engine/synthetic/exam-config.ts` — route observation-recency to the recency
   config path (not the value path).
9. **Golden fixtures:** edit `spike/gen-bundles.mjs` (the four event types → `"observation"` + an
   observation emitter variant that omits `valueQuantity`), then `node spike/gen-bundles.mjs` → regenerates
   the 16 `spike/synthetic/{…}/*.json`.
10. **WebChart adapter cleanup:** the Observation→Procedure synthesis in `normalize.ts` becomes dead for
    these four (their `targetEventType` is now `observation`). Keep the synthesis machinery only if any
    `[Procedure]`-retrieved measure still records as an Observation (none after this) — otherwise remove it
    and `targetEventType`, and update `terminology.ts` comments.
11. **Tests:** update `fhir-bundle-builder.test.ts`, `normalize.test.ts`, `terminology.test.ts`,
    `data-source.test.ts` (invert the synthesis assertions), `codegen-parity.test.ts`,
    `generate-cql*.test.ts`; `cql-execution-engine.test.ts` / `evaluate-measure-cli.test.ts` /
    `value-set-resolver.test.ts` pass automatically iff fixtures + ELM are regenerated consistently.
12. **Docs:** `docs/WEBCHART_FHIR_MAPPING.md` (retire the "Resource-type seam" note — Option A is now done),
    `docs/MEASURES.md`.
13. **Verify:** `pnpm typecheck && pnpm test` green; spot-check the 4 measures still produce
    COMPLIANT/DUE_SOON/OVERDUE/MISSING_DATA across the 4 scenarios (outcomes must be **unchanged**).

Regen order: `gen-measure-bindings` → `gen-cql` → `compile-measures` → `spike/gen-bundles` → test.

## 5. Rollback

Revert the PR — all changes are additive/regenerative (no DB schema, no data migration). The synthesis
workaround returns.

## 6. Recommendation — DEFER (bundle with PR-2c or the E14/standards work)

**Recommend not executing Option A now.** Reasoning:
- **It's a purity gain, not a capability gain.** Real WebChart Observations *already* evaluate these
  measures end-to-end today via the tested synthesis workaround. Option A only removes the workaround.
- **High blast radius vs. low reward:** ~31 files, live measure-semantics change, codegen generalization,
  ELM + 16-fixture regeneration, and the `exam-config` value/recency redesign — a large verification
  surface for a cleanliness win.
- **Better bundled:** the standards-correct destination is option A **and/or** C (VSAC-backed
  `ValueSetResolver`), tied to **E14 PR-3** (blocked on the VSAC key) — and PR-2c will already be in the
  measure/adapter area with the real API to validate against. Doing A then is cheaper and safer.
- **Risk timing:** touching 4 live measures right before the WebChart API contract lands (which may
  reshape the measure/terminology layer anyway) invites rework.

**If we do want it sooner,** the cleanest reduced scope is the two hand-written measures only
(`diabetes_hba1c`, `obesity_bmi`) — they avoid the codegen change entirely — but a partial refactor that
leaves `cholesterol_ldl`/`hypertension` on the synthesis path is arguably worse (two mechanisms for the
same class of measure). So it's all-four-or-defer.

**Proposed disposition:** track Option A as a follow-up on the E14/PR-2c line; keep the working synthesis
until then. (Recorded in `docs/WEBCHART_FHIR_MAPPING.md` §5 as the "standards-correct end state, option A".)
