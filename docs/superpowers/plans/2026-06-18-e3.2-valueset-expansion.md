# E3.2 — Real Value-Set Expansion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make ≥1 value set resolve from a real (store-backed) expansion source and *drive evaluation* through a populated CQL `CodeService`, while the inline-code path stays the default (config-selectable via the engine), proven by cross-mode golden parity on audiogram.

**Architecture:** A `ValueSetResolver` port + a `StoreValueSetResolver` adapter (reads the existing `ValueSetStore`) + `buildCodeService(...)` produce a populated `cql.CodeService`. `CqlExecutionEngine` gains an optional `valueSetResolver` constructor param: with it (and a measure that declares an expansion variant) it evaluates the value-set-retrieve ELM against the populated CodeService; without it (default) it uses today's inline ELM + empty CodeService. Audiogram gets a value-set-retrieve CQL variant compiled to a second ELM; a cross-mode test asserts identical outcomes.

**Tech Stack:** TypeScript via `tsx`; `node:test`/`node:assert`; `cql-execution` (`CodeService`), `cql-exec-fhir`; `@cqframework/cql` build-time compile (`pnpm compile-measures`). No new dependency, no schema change.

**Verified reference facts:**
- `cql.CodeService` (lib/cql-code-service.js): `new cql.CodeService({ [oid]: { [version]: [{code, system, version?}] } })`; a retrieve binds via `findValueSet(vs.id, vs.version)` where `vs.id` is the valueset def's URL. `findValueSetsByOid` returns all versions, so the version key is arbitrary (use `"1"`).
- `ValueSetStore` (`src/stores/value-set-store.ts`): `listAll(): Promise<ValueSetRecord[]>`; `ValueSetRecord = { id, oid, name, …, codes: CodeEntry[] }`; `CodeEntry = { code, display, system }`. No lookup-by-oid method → resolver uses `listAll()` + find on `oid`.
- The `Audiogram Procedures` value set is **already seeded** (`src/measure/value-set-seed.ts`): `oid: "urn:workwell:vs:audiogram-procedures"`, member `{ code: "audiogram-procedure", system: "urn:workwell:vs:audiogram-procedures" }` — exactly the coding `SyntheticFhirBundleBuilder` stamps on the Procedure. **No new seeding needed.**
- `cql-execution-engine.ts`: `constructor()` loads FHIRHelpers; `evaluate()` does `new cql.Library(this.loadElm(meta.library), …)` + `new cql.Executor(library, new cql.CodeService({}), {…})`.
- `measure-registry.ts`: `MEASURES[id] = { id, name, library, periodMonths }`. audiogram → `library: "AnnualAudiogramCompleted-1.0.0"`.
- `audiogram.cql` already declares `valueset "Audiogram Procedures": 'urn:workwell:vs:audiogram-procedures'` but inline-matches the Procedure; the value-set-retrieve form is `[Procedure: "Audiogram Procedures"]`.
- `pnpm compile-measures` compiles every `measures/*.cql` → `src/engine/cql/elm/<LibraryName>-<version>.elm.json` and **regenerates `elm/index.ts`** (auto-generated; idempotent — existing ELMs reproduce with zero diff).
- Engine reads no `process.env` — config is passed via constructor (keeps the worker portable).

**Scope note (recorded):** the deployed runtime stays **inline** (the live runs route keeps `new CqlExecutionEngine()` with no resolver). The expansion mode is delivered as the engine constructor seam + full test coverage incl. cross-mode parity. Wiring a live `WORKWELL_VALUESET_EXPANSION` env toggle into the run pipeline is **deferred** (it would thread a per-request store-backed resolver through the pipeline; not needed to prove the capability and avoids touching the live run path). This is a conscious de-scope from spec §4.2's "env-read at the worker boundary" — the engine *is* config-selectable; only the runtime flip is deferred.

- [ ] **Step 2: Implement `src/engine/cql/value-set-resolver.ts`**

```ts
/**
 * Value-set expansion seam (#90 / E3.2). A pluggable resolver turns a value-set URL into its member
 * codes; `buildCodeService` assembles a populated `cql.CodeService` so a CQL value-set retrieve
 * (`[Procedure: "X"]`) filters by real membership instead of the inline-code workaround. The local
 * `StoreValueSetResolver` reads the existing ValueSetStore; a live VSAC adapter is a future drop-in
 * behind the same port (no engine change).
 */
// eslint-disable-next-line import/no-unresolved
import cql from "cql-execution";
import type { ValueSetStore } from "../../stores/value-set-store.ts";

/** A code as cql-execution's CodeService consumes it. */
export interface CqlCode {
  code: string;
  system: string;
}

/** Expand a value-set URL to its member codes. Returns [] for an unknown/unresolved set (never guesses). */
export interface ValueSetResolver {
  expand(valueSetUrl: string): Promise<CqlCode[]>;
}

/** Local resolver: expands from the ValueSetStore (codes already seeded RESOLVED). */
export class StoreValueSetResolver implements ValueSetResolver {
  constructor(private readonly store: ValueSetStore) {}

  async expand(valueSetUrl: string): Promise<CqlCode[]> {
    const all = await this.store.listAll();
    const vs = all.find((v) => v.oid === valueSetUrl || v.canonicalUrl === valueSetUrl);
    return vs ? vs.codes.map((c) => ({ code: c.code, system: c.system })) : [];
  }
}

/**
 * Build a populated `cql.CodeService` for the given value-set URLs. Shape:
 * `{ [valueSetUrl]: { "1": [{code, system}] } }`. A retrieve resolves via `findValueSet(url)`
 * (version omitted → findValueSetsByOid), so the single "1" version key is sufficient.
 */
export async function buildCodeService(resolver: ValueSetResolver, valueSetUrls: string[]): Promise<unknown> {
  const json: Record<string, Record<string, CqlCode[]>> = {};
  for (const url of valueSetUrls) {
    json[url] = { "1": await resolver.expand(url) };
  }
  return new cql.CodeService(json);
}
```
