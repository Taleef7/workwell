## Executive direction

The self-contained backlog is drained: every remaining big item is MIE-gated, owner-gated, or
research-grade. The bottleneck is no longer engineering — it is MIE engagement. Direction:

- Convert the PoC into an integration-ready product.
- Make the MIE asks explicit — the question package (`docs/MIE_INTEGRATION_QUESTIONS_2026-07-09.md`).
- Pre-build the transport against a mock contract so live integration is days, not weeks, once MIE
  answers land.
- Spend only small, bounded effort keeping the scale story honest — an N=5000 real-evaluation proof,
  and a 1–2 day worker-pool build if the timing warrants it.

**Explicitly not doing right now:** further synthetic-stack polish; a full 120k real-CQL run on Neon
(cost, not value, at this stage); incremental-eval design work ahead of knowing what change signals
WebChart's API actually exposes.

## Decision positions recorded

**Option B (CQL→SQL):** permanently inert unless the triple trigger fires — see #78's updated comment
for the exact conditions. If it is ever built, it is rule-params→SQL codegen for the two rule shapes
(windowed-recency, series-completion), parity-gated against the FHIR-native oracle — never a general
CQL→SQL transpiler.

**E14 literal diff:** ADR-024's "wait for a stable multi-model `@cqframework/cql`" is superseded — the
npm registry (verified 2026-07-09) shows `4.0.0-beta.1` is the only version ever published for that
package, so waiting for it is a dead end. New path: official MADiE eCQM bundles ship pre-compiled ELM
JSON inside `Library.content`, and MITRE's `fqm-execution` (v1.8.5, published 2026-06-03, actively
maintained, same `cql-execution`/`cql-exec-fhir` stack this repo already runs) executes those bundles
directly — no translation needed. See the spike issue (#258) for the gate and plan. Fallback if the
gate fails: run the Java reference translator once at CI build time (build-time JVM does not violate
ADR-008's runtime-JVM-free rule; the classpath-conflict fix is already documented in
`docs/CQF_FHIR_CR_REFERENCE.md` §5).

**Scale honesty:** the claim is "real CQL evaluation of a realistic synthetic population" — the
*evaluation* is real, the *population* is generated. Use exactly this phrasing in demos; never imply
the 120k subjects are real patients.

**worker_threads:** verified viable (2026-07-09 research) — `cql-execution` is stateless pure JS and
safe under `worker_threads`. Work units are subject-index ranges (never live FHIR bundles or
cql-execution objects across the thread boundary); bundles are generated in-worker via the same
`ScaleSubjectGenerator`, which is deterministic on subject index. A hand-rolled pool over
`node:worker_threads` — no new dependency — confined to the batch CLI path only, never the request path.

**Evidence at scale:** tiered by actionability — full `evidence_json` for OVERDUE/DUE_SOON/MISSING_DATA
(load-bearing for cases/worklists), minimal `{scale:true}` for COMPLIANT/EXCLUDED, plus a deterministic
~1% full-evidence sample across all buckets for audit spot-checks. Auto-trim engages above N=20000
subjects unless `--full-evidence` is explicitly passed. Long-term home for large evidence payloads is
the #167 managed-bucket work; Neon keeps status + hash only once that lands.

**Production gaps, ordered** (from the 2026-07-09 strategy): (1) PHI/environment split, (2) auth
(MIE-gated), (3) evidence bucket (#167), (4) observability, (5) scale performance (contract-timed once
MIE answers volume), (6) durable scheduler, (7) real tenancy, (8) backup/DR.
