## 1) The shim — "a simple FHIR server over the WCDB" (3 min)

> Framing: "You asked for a shim — our own FHIR layer over the WebChart SQL database, so the API is
> a swappable contract. Here it is: ~600 lines, plain node:http + one SQL driver, no framework."

```powershell
curl "http://localhost:8085/fhir/Patient?_count=3"          # 56 patients, paged, FHIR R4
curl "http://localhost:8085/fhir/Observation?patient=wc-5"  # real LOINC observations from SQL
```

- Point out: same-origin `link[next]` paging, per-resource `?patient=` composition — the exact
  contract the real WebChart FHIR API serves, verified against MIE's sandbox + teatea.
- **The kicker:** the app consumes it through the same seam as real WebChart — the existing
  56-patient acceptance suite passes against it bucket-for-bucket (`hapi-live.test.ts`).

## 2) The app on live WebChart-schema data (3 min)

Open `http://localhost:3000` → `/compliance` (System = WebChart): the roster shows the 56 dev-DB

`All Systems = Σ tenants` including the `wc` tenant. One sentence: *"WebChart data in, CQL runs our

## 3) CQL→SQL — "boom, the SQL is created" (5 min — the headline)

> Framing: "You asked for CQL→SQL against the WebChart database. We used AI on the CQL + the WCDB
> schema to derive the translations, then locked them as deterministic, reviewed templates — because
> for compliance we need the same SQL every time, not whatever the model says today."

```powershell
Set-Location backend-ts
corepack pnpm generate:sql          # ← the "boom" moment: 4 measures → wcdb-fhir-shim/sql/*.sql
Get-Content ..\wcdb-fhir-shim\sql\hypertension.sql -TotalCount 25
```

Then the simple compliance API Doug described (numerator/denominator + per-patient):

```powershell
curl "http://localhost:8085/compliance/hypertension/cohort?start=2023-06-01&end=2024-06-01"
curl "http://localhost:8085/compliance/wc-5/hypertension?end=2024-06-01"   # "is THIS patient compliant?"
```

**Then the quality argument (this is for Nicole):**

```powershell
Set-Location backend-ts
$env:WCDB_SHIM_PARITY_BASE_URL="http://localhost:8085"
corepack pnpm exec node --import tsx --test src/engine/ingress/webchart/wcdb-sql-parity-live.test.ts
```

> "How do we know the SQL is right? We never trust it on assertion. This gate compares the SQL
> verdict with our CQL engine's verdict **per patient, across all 56, on two evaluation dates** —
> currently zero divergence on 4 measures. A measure that has never passed this gate is never
> served by SQL, and if they ever diverge, we fix the SQL — the CQL engine is the oracle."
> (ADR-025; measures: hypertension, obesity_bmi, diabetes_hba1c, cholesterol_ldl — 27 real
> non-MISSING_DATA outcomes in this dev DB.)

Honest boundaries if asked: windowed-recency measures only (WCDB has no immunization table, so
series SQL can't be parity-proven there); the CMS eCQMs stay on the CQL/fqm path permanently;
the app itself still serves CQL results — SQL is the #292 research track Doug asked to see.

## 4) Codify — LIVE in the app (3 min)

> Framing: "You said our system should be using Codify. It is — as of this week."

Open `http://localhost:3000/studio/cms125` → **Value Sets** tab → **Find a code (Codify)**:

1. Wait for "Offline index ready — 770k entries" (shards stream once from MIE's own
   `ui.mieweb.org/codify` index, then it's fully offline in a Web Worker).
2. Type **"breast cancer screening"** (Doug's exact example) → the top hit is

   authoring; it never writes anything itself).

bundling is the intended path until the npm release ships it (the datavis playbook, ADR-007).

## 5) Fallbacks

| Failure | Move |
|---|---|
| wcdb container won't start | Switch to the HAPI fixtures path (full commands): `docker compose -f infra/docker-compose.yml up -d hapi-fhir`, then from `backend-ts/`: `corepack pnpm load:hapi`, then **repoint + restart the backend** — `$env:WORKWELL_WEBCHART_BASE_URL="http://localhost:8081"; corepack pnpm dev` → identical dashboards; narrate the shim from the PR + this doc |
| Shim dies mid-demo | `docker compose -f infra/docker-compose.yml --profile wcdb restart wcdb-fhir-shim` (or from `wcdb-fhir-shim/`: `npx tsx src/main.ts`); §1 curls are independent of the app |
| Parity run too slow on the call | Pre-captured green output (run it Wed PM, keep the terminal/screenshot); the test is in PR #314 |
| App misbehaves | The CLI tells the same story: `corepack pnpm evaluate:webchart-live --list-patients > r.json` then `corepack pnpm evaluate:webchart-live --roster r.json --date 2024-06-01` |
